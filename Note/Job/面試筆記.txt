''' HTTP 與 Security '''

# 在瀏覽器輸入網址之後的流程
1.瀏覽器會向 DNS 查詢 Domain 對應的 IP
2.瀏覽器會和 Server 建立連線，並將封包送過去
3.目標 Server 接收到封包之後，會根據裡面的 IP 和 PORT 將封包轉送給對應的應用程式
4.進入 MVC 流程

# GET 和 POST 的差異
1.提交網頁表單的時候，GET 會將提交的參數顯示在網址，如果表單有機密資訊不適合用 GET 傳送
> 雖然 GET 也可以將參數藏在 Body 但有的 WebServer 可以讀出數據，有的不行。

2.以 RESTful 的角度而言，GET 用來查詢而 POST 用來新增。

# PUT 和 PATCH 的差異
> PUT 是更新整筆資料，類似 _context.Update(data) 會覆蓋所有的欄位!
> PATCH 是更新部分欄位，常用的修改都是這種類型

# 什麼是 Cookie 和 Session ?
Cookie  : 將用戶狀態儲存在 Client 端
Session : 將用戶狀態儲存在 Server 端
> 不同的用戶擁有不同的Session/Cookie，網站可以透過他們來判斷用戶的各種狀態
> 兩者都不能跨域存取
> Session 物件只要關閉瀏覽器就會被釋放，下次開啟瀏覽器之後會獲得新的 Session 物件
> Cookie 可以設定過期時間，關閉瀏覽器後不一定會失效

# 如何防範 SQL-Injection
> 使用參數化查詢
> 使用 EFCore 搭配 LINQ 他會自動幫你做這件事
var users = context.Users.Where(u => u.Id == userId).ToList();

# 如何防範 XSS
> 對變數內容進行 HTML 編碼，避免 JavaScript 的字段被瀏覽器執行
> 在 cshtml 輸出 CSharp 變數的時候，Razor view engine 會自動幫你做這件事

# 如何解決跨域問題(CORS)
原因 : 瀏覽器的同源策略，會阻擋 ajax 做跨域請求 -> 主要是想防範XSS
解法 : ASP.NET Core MVC/WebApi 有內建處理這個問題的 middleware (只要簡單設定即可)

# 簡述一下 RESTful (GET/POST/PUT/DELETE)
一種 API 的設計風格，透過 HTTP 方法、網址命名、回傳狀態碼，來規範 API 的設計。
GET 成功 -> 回傳 200 並攜帶回傳資料
POST/PUT/DELETE 成功 -> 回傳 204
GET/POST/PUT/DELETE 失敗 -> 回傳 500 並攜帶錯誤訊息

# 說明一下 ViewBag ViewData TempData 的區別
1.ViewBag 和 ViewData 的生命週期僅限於當前請求
2.ViewBag 屬於動態型別，ViewData 屬於 key/value 型別，其中 key 的類型為字串
3.TempData 的生命週期可以跨越多個請求，直到被讀取之後才會銷毀

# 用過哪些 Design Pattern
.NET Core 有內建 DI 框架來實現控制反轉(IOC)，以 EFCore 來舉例
只要將 DbContext 註冊到 DI 容器，就能在 Controller 的建構子添加 DbContext 並透過
它來存取DB，而 Controller 不需要知道 DbContext 是怎麼被 New 出來的，就能降低兩者的依賴。

# 有用過 Vue 嗎?
> 之前無聊的時候，有玩過雙向綁定(v-model)、迴圈渲染(v-for)、搭配 axios 打後端 api

# 描述一下DI注入的三種生命週期
Singleton : 只會產生一個 Instance 並永遠共用

Transient : 每個注入點都會建立新的實例，永遠不會共用。
假設這次的請求需要兩個 Service 處理商業邏輯，且這兩個 Service 都會注入 DbContext，
那麼這兩個 Service 拿到的 DbContext 不會是同一個 Instance

Scoped : 在一個請求結束之前，共用一個 Instance 
如果有多個 Service 都需要注入"類別A"，那這些"類別A"都會是同一個 Instance

# 物件的基本三大特性 -> 封裝 & 繼承 & 多型
封裝 : 隱藏實作細節，僅提供函數給外部存取。
繼承 : 讓子類別可以 reuse 父類別的程式碼(用途舉例 : 繼承 Filter 或分頁參數)
多型 : 例如 Override (子類別可以覆寫父類別的 Virtual function)
> 實務上我比較常用的是繼承，例如有一個 BaseController 頭上會有很多 Filter
> 那其他的 Controller 如果也需要冠上這些 Filter 就可以繼承 BaseController

# 什麼是 MVC ?
MVC是一種程式架構，它將程式碼切成MVC三個部分，目的是讓程式碼更容易維護和擴充。
C : Controller -> 接收 User 送來的請求
M : Model -> 處理商業邏輯
V : View  -> 呈現資料處理的結果

User 的請求會先進到 Controller，然後再送到 Model 裡面做運算，
最後再透過 View 來呈現處理結果。

# 解釋一下什麼是 virtual function (C#)
可以讓子類別 override 的 function，是一種多型的概念。
子類別會繼承這個 virtual function 的實作，如果沒有 override 則會調用父類別的實作。

# 解釋一下 CSharp 變數或類別的存取範圍
private  : 只有類別本身可以存取
protect  : 只有類別本身和子類別可以存取
public   : 任何地方都可以存取
internal : 同一個dll或exe檔案

# EF 內建 Cache 機制的特點
1.這個機制預設就會啟動，會將查詢結果暫存(Change Tracking)以提高效能
2.這個機制只對同一個 DbContext 有效(在同一個 DbContext 使用相同的查詢，會抓到相同的結果)
> 如果要在同一個 DbContext 使用相同的查詢，並且確保撈取出最新結果，必須使用 AsNoTracking
> AsNoTracking 只能用在查詢，其撈出來的資料不會被追蹤變更

# 使用 ID 和 GUID 當 PK 有什麼優缺點
1.使用 ID 當流水號，就不用在新增的時候賦值給ID
> 使用 GUID 則必須自行賦值，否則預設為 Guid.Empty (00000000-0000-0000-0000-000000000000)

2.使用 GUID 的好處是永遠不會重複，適合用在分散式系統來實現資料的唯一性。

# ORM
將DB資料映射到物件的一種技術，
優點是令開發人員不需要撰寫SQL就可以存取DB，
缺點是使用複雜語法的效能可能會比較差(透過 ORM 轉出來的 SQL 可能效率不好)，
如果有一段查詢的語法非常複雜，而且需要優化到極致，可能就不適合使用 ORM !

# EFCore 兩種資料加載的技術 -> Eager loading 與 Lazy loading 的差異
> 預設採用 Lazy loading 而不會載入關聯的資料
> 若沒有在後面接 ToList() / Count() / FirstOrDefault() 之類的，則 EFCore 不會去撈資料
var orders = context.Orders(o => o.Id == 1).ToList(); // 不會去撈 OrderDetails

> 示範改用 Eager loading 來一次性載入訂單與明細
> Include 是告訴 EFCore 要連同關聯的資料一起撈取，但如果沒有接 ToList() 也不會去撈資料
var orders = context.Orders.Include(o => o.OrderDetails).ToList();

# 為什麼 DbContext 的生命週期預設是 Scoped 而不是 Transient ?
> 為了保證在一個請求之中，多個 Service 使用的 DbContext 是同一個 Instance
> 避免發生資料庫鎖定，造成另一個 DbContext 存取出錯。

# 為什麼 DbContext 的生命週期預設是 Scoped 而不是 Singleton ?
這樣會造成 Connection pool 裡面的連線資源無法立即被釋放(預設的 timeout 是 30 秒)。
